🧪 Edge Case Testing

Edge Case — Single Order Dispatch

Create 1 new PENDING order, then trigger dispatch. Should succeed with stop_count=1 and NO ORS API call in server logs.

Steps:
  1. Create a new zone (or use existing one) with min_order_count = 1
  2. Add 1 PENDING order in that zone
  3. Update cutoff_time to past: UPDATE delivery_zones SET cutoff_time = '00:01:00', min_order_count = 1 WHERE id = 'zone-id';
  4. GET dispatch/ready → note batch_id
  5. Assign EMP → Dispatch
  6. Verify: stop_count=1, no ORS warning in server terminal

Edge Case — ORS Fallback (Bad API Key)

Test that dispatch still succeeds when ORS fails.

Steps:
  1. In backend/.env, change to: ORS_API_KEY=bad_key_12345
  2. Restart the server
  3. Create new test data (5+ orders, new batch)
  4. Run the dispatch flow
  5. Verify: returns 200, server logs show "ORS optimisation failed, using original order"
  6. batch_stops still created, batch status = DISPATCHED
  7. Restore real ORS_API_KEY and restart server after testing!

Edge Case — Inactive Employee Login

Test 401 for inactive delivery staff.

Steps:
  1. Run in DBeaver: UPDATE delivery_staff SET is_active = false WHERE emp_id = 'EMP001';
  2. Try delivery login with EMP001 → should get 401 "Employee account is inactive"
  3. Restore: UPDATE delivery_staff SET is_active = true WHERE emp_id = 'EMP001';

Valkey (Redis) Verification Commands

Run these in your terminal to verify Valkey state.

After Dispatch:
redis-cli GET emp:{emp_uuid}:active_batch → returns the batch_id UUID string
redis-cli TTL emp:{emp_uuid}:active_batch → returns ~86400

After Delivery Login:
redis-cli GET session:{session_token} → returns the emp_id string (e.g. EMP001)
redis-cli TTL session:{session_token} → returns ~43200 (12h)

After All Stops Completed:
redis-cli GET emp:{emp_uuid}:active_batch → returns (nil) — batch cleared
